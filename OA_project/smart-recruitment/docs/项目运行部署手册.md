# 企业智慧招聘 OA 软件项目运行部署手册

## 一、文档定位

本文档用于指导企业智慧招聘 OA 系统在开发机、联调环境和服务器环境中运行部署。所有命令默认以 `smart-recruitment` 为工程根目录，不使用本机绝对路径；实际部署时可将工程放在任意目录，只需先进入工程根目录再执行命令。

系统采用前后端分离架构：

| 模块 | 技术栈 | 默认端口 | 说明 |
| --- | --- | --- | --- |
| `recruitment-web` | Vue 3 + Vite | 3000 | 前端页面 |
| `recruitment-service` | Spring Boot 3.2 | 8080 | 核心招聘业务服务 |
| `recruitment-ai-service` | Spring Boot 3.2 | 8081 | AI 简历解析、匹配、面试服务 |
| MySQL | 8.0 | 3306 | 业务数据库 |
| Redis Stack | latest | 6379 / 8001 | 缓存与向量能力预留 |
| Elasticsearch | 7.17.21 | 9200 | 简历和职位全文检索 |
| Nacos | 2.2.3 | 8848 / 9848 | 配置中心与服务注册发现 |

推荐启动顺序：`MySQL / Redis / Elasticsearch / Nacos` -> `recruitment-ai-service` -> `recruitment-service` -> `recruitment-web`。

## 二、部署方式选择

| 场景 | 推荐方式 | 特点 |
| --- | --- | --- |
| 前端页面开发 | 启动 `recruitment-web` + 核心服务 | 前端当前强制关闭 Mock，会直连真实后端 |
| 本地轻量后端开发 | `local` profile | 需要 MySQL，不依赖 Nacos、Redis、Elasticsearch |
| 完整本地联调 | Docker Compose 中间件 + `dev` profile | 最接近团队联调环境，依赖 Nacos 配置中心 |
| 服务器部署 | 中间件独立部署 + 后端 jar + 前端静态资源 | 需按服务器网络设置环境变量和反向代理 |

特殊性说明：

- `dev` profile 会优先读取 Nacos 配置；Nacos 配置缺失时会告警并回退到本地 `application-dev.yml`。
- `local` profile 不启用 Nacos、Redis、Elasticsearch，但仍需要可访问的 MySQL。
- 核心服务默认通过 Nacos 服务发现调用 `recruitment-ai-service`；只有设置 `AI_SERVICE_URL` 时才改为直连。
- 前端开发环境通过 Vite 代理转发 `/api`，不能直接打开 `index.html` 进行联调。

## 五、配置准备

### 5.1 环境变量文件

工程提供 `.env.example`，用于 Docker Compose 和人工配置参考：

```bash
cp .env.example .env
```

Windows PowerShell：

```powershell
Copy-Item .env.example .env
```

注意：

- `.env` 已被 Git 忽略，不要提交真实密码、生产密钥或个人 API Key。
- Docker Compose 会读取 `--env-file .env`。
- Spring Boot 不会自动读取 `.env`，后端进程需要通过 shell、IDE、系统服务或容器环境变量显式注入。

### 5.2 关键环境变量

| 变量 | 默认值 | 使用方 | 说明 |
| --- | --- | --- | --- |
| `DB_HOST` | `localhost` | 核心服务 | MySQL 地址 |
| `DB_PORT` | `3306` | 核心服务 | MySQL 端口 |
| `DB_NAME` | `smart_recruitment` | 核心服务 | 数据库名 |
| `DB_USERNAME` | `root` | 核心服务 | 数据库账号 |
| `DB_PASSWORD` | `root` | 核心服务 / Docker | 数据库密码 |
| `REDIS_HOST` | `localhost` | 核心服务 | Redis 地址 |
| `REDIS_PORT` | `6379` | 核心服务 | Redis 端口 |
| `ES_HOST` | `127.0.0.1` | 核心服务 | Elasticsearch 地址 |
| `ES_PORT` | `9200` | 核心服务 | Elasticsearch 端口 |
| `NACOS_SERVER_ADDR` | `127.0.0.1:8848` | 两个后端服务 | Nacos 地址 |
| `NACOS_CONFIG_GROUP` | `SMART_RECRUITMENT` | 两个后端服务 | Nacos 配置分组 |
| `NACOS_DISCOVERY_GROUP` | `SMART_RECRUITMENT` | 两个后端服务 | Nacos 注册分组 |
| `NACOS_NAMESPACE` | 空 | 两个后端服务 | Nacos 命名空间 ID |
| `NACOS_DISCOVERY_IP` | 空 | 两个后端服务 | 多网卡环境下显式注册地址 |
| `AI_SERVICE_URL` | 空 | 核心服务 | Nacos 配置已加载且为空时走服务发现；本地回退配置默认直连 AI 服务 |
| `OPENAI_API_KEY` | 空 | AI 服务 | OpenAI 兼容接口 Key |
| `OPENAI_BASE_URL` | `https://api.openai.com` | AI 服务 | OpenAI 兼容接口地址 |
| `OPENAI_MODEL` | `qwen3.7-plus` | AI 服务 | 模型名称 |
| `FILE_UPLOAD_DIR` | `uploads` | 核心服务 | 上传文件目录，建议保持相对路径 |

PowerShell 设置示例：

```powershell
$env:DB_PASSWORD="root"
$env:OPENAI_API_KEY="your-api-key"
```

Bash 设置示例：

```bash
export DB_PASSWORD=root
export OPENAI_API_KEY=your-api-key
```

## 六、中间件部署

### 6.1 Docker Compose 方式

适用于本地完整联调。该方式会启动 MySQL、Redis Stack、Elasticsearch、Nacos，并通过 `nacos-config-init` 发布 Nacos 配置。`nacos-config-init` 会等待 Nacos 健康后再发布配置。

```bash
docker compose --env-file .env -p recruitment -f docker/docker-compose.yml up -d
```

查看状态：

```bash
docker compose --env-file .env -p recruitment -f docker/docker-compose.yml ps
```

重新发布 Nacos 配置：

```bash
docker compose --env-file .env -p recruitment -f docker/docker-compose.yml up --force-recreate nacos-config-init
```

停止中间件：

```bash
docker compose --env-file .env -p recruitment -f docker/docker-compose.yml down
```

如需重建数据库并重新执行 `sql/init.sql`：

```bash
docker compose --env-file .env -p recruitment -f docker/docker-compose.yml down
docker volume rm recruitment_mysql-data
docker compose --env-file .env -p recruitment -f docker/docker-compose.yml up -d mysql
```

### 6.2 MySQL

Docker 首次创建数据卷时会自动执行：

```text
sql/init.sql
```

手动初始化方式：

```bash
mysql -u root -p
```

```sql
CREATE DATABASE IF NOT EXISTS smart_recruitment DEFAULT CHARSET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE smart_recruitment;
SOURCE sql/init.sql;
```

初始化脚本会创建默认管理员账号：

| 用户名 | 密码 | 角色 |
| --- | --- | --- |
| `admin` | `admin123` | 管理员 |

### 6.3 Nacos

本项目 `dev` profile 会优先读取以下 Data ID，默认分组为 `SMART_RECRUITMENT`：

| Data ID | 内容 |
| --- | --- |
| `recruitment-common.yml` | JWT、安全策略、公共日志配置 |
| `recruitment-service.yml` | 核心服务端口、MySQL、Redis、ES、上传目录 |
| `recruitment-ai-service.yml` | AI 服务端口、模型地址、模型参数 |

本地 Docker Nacos 默认未启用鉴权，`NACOS_USERNAME` 和 `NACOS_PASSWORD` 保持为空。若服务器 Nacos 启用了鉴权，需要在后端进程和配置发布流程中设置对应变量。

命名空间注意事项：

- `NACOS_NAMESPACE` 填写命名空间 ID，不是显示名称。
- 本地默认空命名空间即可。
- 多环境部署时建议按 `dev`、`test`、`prod` 分命名空间或分集群隔离。

### 6.4 Redis Stack

连通性检查：

```bash
docker exec -it recruitment-redis-stack redis-cli ping
```

预期返回：

```text
PONG
```

如果只使用 `local` profile，可暂不启动 Redis。

### 6.5 Elasticsearch

连通性检查：

```bash
curl http://localhost:9200
```

能返回版本和集群信息即表示可用。Elasticsearch 启动较慢，首次启动后建议等待一段时间再启动核心服务。

## 七、后端运行

### 7.1 首次构建

```bash
cd recruitment-backend
mvn clean install -DskipTests
```

如需运行测试：

```bash
mvn test
```

### 7.2 本地轻量模式

适用于只验证核心接口或前后端基础联通。

特点：

- 使用 `local` profile。
- 不读取 Nacos。
- 不启用 Redis 和 Elasticsearch 自动配置。
- 仍需要 MySQL 可访问。
- 核心服务直连 AI 服务，默认地址为 `http://127.0.0.1:8081`。

启动 AI 服务：

```bash
cd recruitment-backend/recruitment-ai-service
mvn spring-boot:run "-Dspring-boot.run.profiles=local"
```

启动核心服务：

```bash
cd recruitment-backend/recruitment-service
mvn spring-boot:run "-Dspring-boot.run.profiles=local"
```

如果 AI 服务未启动，涉及 AI 的功能会失败；非 AI 的职位、简历、登录等功能仍可单独调试。

### 7.3 完整联调模式

适用于验证 Nacos、Redis、Elasticsearch、AI 服务调用和完整链路。

前置条件：

- Docker Compose 或独立中间件已启动。
- 建议 Nacos 已发布 `nacos/config/*.yml`；未发布时会回退到本地 `application-dev.yml` 并输出告警。
- MySQL 已初始化 `smart_recruitment` 数据库。
- AI 服务已配置 `OPENAI_API_KEY`，否则真实 AI 调用会失败。

启动 AI 服务：

```bash
cd recruitment-backend/recruitment-ai-service
mvn spring-boot:run "-Dspring-boot.run.profiles=dev"
```

启动核心服务：

```bash
cd recruitment-backend/recruitment-service
mvn spring-boot:run "-Dspring-boot.run.profiles=dev"
```

Nacos 服务发现检查：

- Nacos 控制台可访问 `http://localhost:8848/nacos`。
- 如果使用 Nacos 服务发现调用 AI 服务，服务列表中应存在 `recruitment-service` 和 `recruitment-ai-service`。
- 两个服务实例均为健康状态。

### 7.4 jar 包方式运行

服务器部署通常使用 jar 包运行：

```bash
cd recruitment-backend
mvn clean package -DskipTests
```

启动 AI 服务：

```bash
java -jar recruitment-ai-service/target/recruitment-ai-service-1.0.0.jar --spring.profiles.active=dev
```

启动核心服务：

```bash
java -jar recruitment-service/target/recruitment-service-1.0.0.jar --spring.profiles.active=dev
```

生产或测试环境建议通过系统服务、进程管理器或容器平台托管 jar 进程，并由部署平台注入环境变量。不要依赖开发机 shell 中临时设置的变量。

### 7.5 AI 服务密钥

AI 服务支持 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL` 等变量。仓库中的 `application-secrets.properties` 仅适合团队开发环境，不应作为生产部署依据。

生产部署建议：

- 使用环境变量、密钥管理服务或配置平台注入 Key。
- 不在 Git 中提交生产 Key。
- 如果运行目录变化，不依赖相对位置读取本地 secret 文件，优先使用环境变量。

## 八、前端运行与构建

### 8.1 开发运行

```bash
cd recruitment-web
npm install
npm run dev
```

默认访问地址：

```text
http://localhost:3000
```

当前 Vite 代理规则：

| 前端请求 | 转发目标 |
| --- | --- |
| `/api/ai/mock-interview/**` | `http://localhost:8081` |
| `/api/**` | `http://localhost:8080` |

当前 `src/utils/request.js` 中 `FORCE_DISABLE_MOCK = true`，前端会强制关闭 Mock 并直连真实后端。若要恢复纯前端 Mock 演示，需要修改该常量并重新启动前端开发服务。

### 8.2 生产构建

```bash
cd recruitment-web
npm install
npm run build
```

构建产物位于：

```text
recruitment-web/dist
```

部署时将 `dist` 内容交给 Nginx、静态资源服务器或对象存储托管，并将 `/api` 反向代理到核心服务。若模拟面试接口继续由 AI 服务直接承接，需要为 `/api/ai/mock-interview` 单独代理到 AI 服务；否则可统一经过核心服务网关或后端入口。

反向代理核心要点：

```nginx
location / {
    try_files $uri $uri/ /index.html;
}

location /api/ai/mock-interview {
    proxy_pass http://recruitment-ai-service:8081;
}

location /api {
    proxy_pass http://recruitment-service:8080;
}
```

服务器环境中的 `recruitment-ai-service`、`recruitment-service` 可替换为实际 DNS、容器服务名或内网主机名。

## 十一、服务器部署建议

服务器部署和本地联调的主要区别在网络、配置和进程托管：

| 项 | 本地联调 | 服务器部署 |
| --- | --- | --- |
| 中间件地址 | 常用 `localhost` | 使用内网域名、服务名或独立主机名 |
| 配置来源 | `.env` + Nacos 模板 | 配置中心、系统环境变量、密钥平台 |
| 前端入口 | Vite dev server | 静态资源服务器 + 反向代理 |
| 后端运行 | Maven 插件 | jar、系统服务、容器或编排平台 |
| 文件上传 | `uploads` | 挂载持久化目录，并保持应用内配置可迁移 |

部署建议：

- `dev`、`test`、`prod` 使用不同 Nacos 命名空间或不同 Nacos 集群。
- 生产环境必须替换 `JWT_SECRET`、数据库密码和模型 API Key。
- `NACOS_DISCOVERY_IP` 只在多网卡、容器桥接网络、跨主机注册异常时设置，值必须能被其他服务访问。
- 后端服务和中间件不在同一主机时，显式设置 `DB_HOST`、`REDIS_HOST`、`ES_HOST`、`NACOS_SERVER_ADDR`。
- 上传目录保持相对配置或由部署平台映射，不在文档和代码中写死机器路径。
- 前端对外域名、后端服务内网地址、Nacos 地址应分层管理，避免浏览器访问内网服务名。

## 十二、常见问题排查

### 12.1 端口被占用

现象：

```text
Port 8080 was already in use
EADDRINUSE: address already in use
```

处理：

- 找到占用 `3000`、`8080`、`8081`、`8848`、`9200`、`3306` 的进程并停止。
- 或修改对应端口：后端为 `server.port` / `RECRUITMENT_SERVICE_PORT` / `RECRUITMENT_AI_SERVICE_PORT`，前端为 `recruitment-web/vite.config.js`。

### 12.2 MySQL 连接失败

常见原因：

- MySQL 未启动。
- `DB_HOST`、`DB_PORT`、`DB_USERNAME`、`DB_PASSWORD` 与实际环境不一致。
- 数据库 `smart_recruitment` 未创建或 `sql/init.sql` 未执行。
- Docker 数据卷已存在，初始化 SQL 不会重复自动执行。

检查：

```bash
docker exec -it recruitment-mysql mysql -uroot -proot -e "SHOW DATABASES;"
```

### 12.3 Nacos 连接或配置导入异常

常见原因：

- Nacos 未启动或端口不可达，日志可能出现 `Fail to connect server`。
- `nacos-config-init` 未成功发布配置。
- `NACOS_CONFIG_GROUP`、`NACOS_NAMESPACE` 与实际 Data ID 所在位置不一致。
- 启用了鉴权但未设置 `NACOS_USERNAME`、`NACOS_PASSWORD`。

处理：

```bash
docker compose --env-file .env -p recruitment -f docker/docker-compose.yml up --force-recreate nacos-config-init
```

当前 `dev` profile 允许回退到本地配置。若日志只有 Nacos 连接告警但后端随后正常启动，说明走了本地回退；若需要验证完整 Nacos 链路，应先确认 `8848` 和 `9848` 都可访问，并确认 3 个 Data ID 已发布。

### 12.4 Redis 或 Elasticsearch 连接失败

处理：

```bash
docker exec -it recruitment-redis-stack redis-cli ping
curl http://localhost:9200
```

如果只做轻量开发，可使用 `local` profile 暂时绕过 Redis 和 Elasticsearch。

### 12.5 前端请求 404 或代理不生效

处理：

- 确认通过 `npm run dev` 启动前端。
- 确认核心服务 `8080` 已启动。
- 确认 `recruitment-web/vite.config.js` 中 `/api` 代理目标与后端地址一致。
- 生产环境确认反向代理包含 SPA 回退规则和 `/api` 转发规则。

### 12.6 登录后 401

可能原因：

- Token 过期。
- 浏览器保留了旧登录态。
- 后端 `JWT_SECRET` 已变更。

浏览器控制台清理：

```javascript
localStorage.removeItem('token')
localStorage.removeItem('refreshToken')
localStorage.removeItem('userInfo')
location.reload()
```

### 12.7 AI 调用失败

常见原因：

- `recruitment-ai-service` 未启动。
- `OPENAI_API_KEY` 未设置或无效。
- 模型服务地址不可达。
- 核心服务没有通过 Nacos 找到 AI 服务，或 `AI_SERVICE_URL` 指向错误。

处理：

- 确认 AI 服务端口可访问。
- 在 Nacos 服务列表中确认 `recruitment-ai-service` 健康。
- 需要直连时设置 `AI_SERVICE_URL`。
- 检查 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL`。

### 12.8 文件上传失败

常见原因：

- 应用运行目录无写权限。
- `FILE_UPLOAD_DIR` 指向不可写目录。
- 文件超过默认限制。

默认上传目录为：

```text
uploads
```

配置项：

```yaml
spring:
  servlet:
    multipart:
      max-file-size: 10MB
      max-request-size: 10MB
file:
  upload-dir: ${FILE_UPLOAD_DIR:uploads}
  max-size: ${FILE_MAX_SIZE:10485760}
```
