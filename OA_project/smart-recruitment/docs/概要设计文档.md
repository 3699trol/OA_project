## 1. 概述

### 1.1 项目背景与目标

本项目是面向企业招聘场景的智能化招聘平台，覆盖职位发布、简历管理、投递流转、面试安排、人岗智能匹配、统计分析等核心业务，并集成 Elasticsearch 全文检索、Redis Stack 向量检索与 Spring AI 大模型能力，实现"AI + 招聘"的智能化闭环。

### 1.2 设计目标

- **前后端分离**：前端 Vue 3 SPA，后端 Spring Boot 提供 REST API。
- **业务与 AI 解耦**：核心招聘业务（`recruitment-service`）与 AI/向量能力（`recruitment-ai-service`）拆分为两个独立微服务，通过 OpenFeign 调用。
- **角色权限清晰**：管理员 / HR / 面试官 / 求职者四类角色，权限边界明确（尤其正式面试题与模拟面试接口隔离）。
- **可插拔中间件**：本地开发可脱离 Nacos/MySQL/Redis/ES 独立运行（`local` Profile），联调环境统一通过 Docker Compose 拉起中间件（`dev` Profile）。

### 1.3 名词说明

| 术语 | 说明 |
|------|------|
| Result\<T\> | 后端统一响应包装体 |
| Embedding | 简历/职位文本向量化后的浮点数组，用于向量相似度检索 |
| ai_match_score | AI 计算的人岗匹配得分 |

### 2.3 部署架构

- **本地轻量模式**（`local` Profile）：仅启动 `recruitment-service` / `recruitment-ai-service` 两个 Spring Boot 进程 + 前端 Vite 开发服务器，Nacos/Redis/ES/MySQL 全部禁用，用于前后端联通验证。
- **完整联调模式**（`dev` Profile）：通过 `docker/docker-compose.yml` 一次性拉起 `mysql` / `redis-stack` / `elasticsearch` / `nacos` 四个容器，后端两个服务分别以 `dev` Profile 启动并注册进 Nacos，配置支持从 Nacos Config 动态拉取。
- **端口规划**：前端 `3000`；核心服务 `8080`；AI 服务 `8081`；Nacos 控制台 `8848`；ES `9200`；Redis `6379`/RedisInsight `8001`；MySQL `3306`。

## 3. 模块拆分设计

### 3.1 工程模块划分（Maven 多模块）

```text
recruitment-backend/
├── recruitment-common/          # 公共能力（被 service / ai-service 依赖）
│   ├── common-core              # Result、PageResult、BusinessException、错误码、工具类、UserContext
│   ├── common-web                # 全局异常处理、参数校验、操作日志注解、请求拦截器
│   ├── common-security           # Spring Security 配置、JWT 生成解析、认证过滤器、权限校验
│   └── common-redis              # RedisTemplate 配置、缓存工具、分布式锁、防重复提交、Token 黑名单
├── recruitment-api/              # 服务间 DTO + Feign 客户端接口（AiServiceClient 等）
├── recruitment-service/          # 核心招聘业务服务（端口 8080）
└── recruitment-ai-service/       # AI 与向量服务（端口 8081）
```

依赖关系：`recruitment-service` 与 `recruitment-ai-service` 均依赖 `recruitment-common` 与 `recruitment-api`；`recruitment-service` 通过 `recruitment-api` 中定义的 Feign 接口调用 `recruitment-ai-service`，避免两服务重复定义请求/响应对象。

### 3.2 核心服务（recruitment-service）业务模块

| 模块 | 职责 | 关联数据表 |
|------|------|-----------|
| `system` | 用户/角色/权限管理，用户启用禁用 | `sys_user`、`sys_role`、`sys_permission`、`sys_user_role` |
| `auth` | 注册、登录、退出、当前用户、Token 刷新、改密 | `sys_user` |
| `job` | 职位 CRUD、发布/下架、职位向量同步触发 | `job` |
| `resume` | 简历编辑、文件上传、AI 解析/评估调用、ES/向量同步触发 | `resume`、`resume_education`、`resume_experience` |
| `application`（对应"投递"，非 Spring Application） | 投递、防重复投递、HR 初筛、状态流转、流转日志 | `job_application`、`application_log` |
| `interview` | 面试创建与分配、AI 出题调用、面试评价 | `interview`、`interview_evaluation` |
| `search` | ES 简历/职位检索、高亮、索引重建 | ES `job` / `resume` 索引 |
| `dashboard` | HR 工作台统计（职位数/投递数/待筛选/面试数/转化率） | 汇总查询，无独立表 |
| `file` | PDF/Word 上传下载、Apache Tika 文本提取 | `resume.file_url` |

每个业务模块统一分层：`controller → service → mapper → entity`，并配套 `dto`（入参）与 `vo`（出参），禁止 Entity 直接暴露给前端。

### 3.3 AI 服务（recruitment-ai-service）业务模块

| 模块 | 职责 |
|------|------|
| `resume` | 简历结构化解析、质量评估与优化建议，固定 JSON 输出解析 |
| `matching` | 简历-职位向量匹配、技能/学历/年限综合评分、匹配理由生成 |
| `question` | 正式面试题生成（仅面试官可调用），根据简历+职位生成追问题 |
| `mockinterview` | 求职者模拟面试题生成、追问、报告生成（与正式面试题接口/权限隔离） |
| `evaluation` | 模拟面试回答评估、辅助面试官整理总结 |
| `vector` | 生成职位/简历 Embedding，写入 Redis Stack，相似度检索 |
| `config` | Spring AI 模型配置、Embedding 模型配置、Redis Stack 配置、提示词与超时/重试配置 |

## 4. 接口整体规范

### 4.1 通用约定

- **统一前缀**：所有接口路径以 `/api` 开头。
- **统一响应体**：`Result<T>`，包含状态码、消息、数据，禁止裸返回业务对象（文件流除外）。
- **统一鉴权**：`Authorization: Bearer <JWT>`；默认所有接口需登录，白名单仅限注册/登录等公开接口。
- **统一异常**：业务异常通过 `BusinessException` + 错误码枚举抛出，由 `common-web` 全局异常处理器统一转换为 `Result`。
- **分页参数**：统一 `PageResult` 封装，查询接口约定 `pageNum`/`pageSize` 风格参数。
- **操作日志**：关键写操作（发布职位、修改投递状态、面试评价等）通过操作日志注解记录，敏感信息脱敏。

### 4.2 接口分组

| 分组 | 前缀 | 说明 |
|------|------|------|
| 认证 | `/api/auth/**` | 注册/登录/登出/当前用户/刷新Token/改密 |
| 系统管理 | `/api/system/**` | 用户/角色/权限 |
| 职位 | `/api/job/**` | 职位 CRUD、发布/下架 |
| 简历 | `/api/resume/**` | 简历 CRUD、AI 解析/评估 |
| 投递 | `/api/application/**` | 投递、状态流转 |
| 面试 | `/api/interview/**` | 面试创建、出题、评价 |
| 搜索 | `/api/search/**` | ES 职位/简历搜索、索引重建 |
| AI | `/api/ai/**` | 简历解析、人岗匹配、出题、模拟面试（AI 服务侧，经核心服务 Feign 转发或前端直连） |
| 统计 | `/api/dashboard/**` | HR 工作台统计 |
| 文件 | `/api/file/**` | 上传/下载 |

详细接口清单见 `docs/api-docs.md`。

### 4.3 权限边界规范

- **正式面试题**（`/api/interview/question/generate`、AI 服务 `/api/ai/question/generate`）仅面试官角色可调用。
- **模拟面试**（AI 服务 `/api/ai/mock-interview/**`）仅求职者角色可调用，与正式出题接口物理隔离（不同 Controller/不同权限校验）。
- 方法级权限基于 Spring Security + 自定义权限注解，按角色（管理员/HR/面试官/求职者）与数据权限（如 HR 仅能操作本公司职位）双重校验。

### 4.4 服务间调用规范

- `recruitment-service` 调用 `recruitment-ai-service` 统一通过 `recruitment-api` 模块中定义的 `AiServiceClient`（OpenFeign 接口），请求/响应对象集中定义为 DTO（如 `AiResumeParseRequest/Response`、`AiMatchRequest/Response`），避免散落重复定义。
- 本地轻量模式下 AI 服务地址通过配置项 `ai.service.url`（默认 `http://localhost:8081`）直连；完整模式下由 Nacos 服务发现解析。

## 5. 数据库概要设计

### 5.1 设计规范

- 统一下划线命名（`job_application.status`）。
- 主数据表统一包含 `create_time`、`update_time`（`DEFAULT CURRENT_TIMESTAMP` + `ON UPDATE`）。
- 逻辑删除统一使用 `deleted TINYINT DEFAULT 0`。
- 金额 `DECIMAL(10,2)`，评分 `DECIMAL(5,2)`/`INT`，布尔 `TINYINT(1)`。

### 5.2 核心表清单与关系

```text
sys_user ──< company (company_id)
sys_user ──< job.publisher_id (HR发布)
sys_user ──< resume.user_id (求职者)
job ──< job_application.job_id
sys_user ──< job_application.user_id
resume ──< job_application.resume_id
job_application ──< application_log.application_id
job_application ──< interview.application_id
sys_user ──< interview.interviewer_id
interview ──< interview_evaluation.interview_id
resume ──< resume_education.resume_id (1:N, CASCADE)
resume ──< resume_experience.resume_id (1:N)
```

| 表名 | 说明 | 关键索引/约束 |
|------|------|---------------|
| `sys_user` | 用户主表（含四类角色 `user_type`） | `uk_sys_user_username`、`idx_sys_user_phone`、`idx_sys_user_company` |
| `company` | 公司信息 | — |
| `job` | 职位信息 | `idx_job_status`、`idx_job_city`、全文索引 `ft_job_content` |
| `resume` | 简历主表，含 `embedding_vector`（Redis Stack 向量备份）、`parsed_content`（供 ES 索引） | — |
| `resume_education` | 教育经历子表 | 外键 `resume_id` + `ON DELETE CASCADE` |
| `resume_experience` | 工作经历子表 | — |
| `job_application` | 投递记录，含 AI 匹配得分/理由 | `uk_job_user (job_id, user_id)` 防重复投递、`idx_application_status` |
| `application_log` | 投递状态流转日志 | — |
| `interview` | 面试安排，含 AI 出题缓存 | — |
| `interview_evaluation` | 面试评价（技术/沟通/逻辑/综合评分） | — |

> 完整字段定义见 `sql/init.sql` 与 `README.md` 第 4 章。

### 5.3 多数据存储协同一致性

简历/职位数据在写入 MySQL 后，需按以下顺序保持三方一致：

1. **MySQL**：写入/更新主数据（事务边界）。
2. **Elasticsearch**：同步 `parsed_content`/职位描述等文本，供全文检索。
3. **Redis Stack**：同步 Embedding 向量，供相似度检索。

同步失败需有补偿机制（如异步重试队列或索引重建接口 `/api/search/rebuild-index`），避免三端数据漂移。

## 6. 第三方组件集成设计

### 6.1 Nacos（服务注册与配置中心）

- **依赖**：Spring Cloud Alibaba Nacos（`spring-cloud-starter-alibaba-nacos-discovery`、`-config`）。
- **本地模式**（`application.yml`，默认 `local` Profile）：`spring.cloud.nacos.discovery.enabled=false`、`config.enabled=false`，服务不注册、不拉取配置，纯本地运行。
- **联调模式**（`application-dev.yml`）：
  - `discovery.server-addr=localhost:8848`，`fail-fast=false`（Nacos 不可用时不阻塞启动）。
  - `config.server-addr=localhost:8848`，`file-extension=yml`，通过 `spring.config.import: optional:nacos:${spring.application.name}.yml` 动态导入远程配置，实现配置集中管理与热更新。
- **容器版本**：`nacos/nacos-server:v2.2.3`，`MODE=standalone`。
- **两个服务**（`recruitment-service`、`recruitment-ai-service`）均以 `spring.application.name` 作为服务名注册，供 OpenFeign 按服务名调用（联调模式下）。

### 6.2 Redis / Redis Stack

- **职责划分**：
  - 基础 Redis 能力（缓存、Token 黑名单、验证码、分布式锁、防重复提交）由 `common-redis` 模块统一封装 `RedisTemplate`。
  - **Redis Stack**（`redis/redis-stack:latest`）额外提供向量检索能力，用于存储简历/职位 Embedding，支持相似度检索（`recruitment-ai-service` 的 `vector` 模块）。
- **配置**：`application-dev.yml` 中 `spring.data.redis.host/port` 指向 `localhost:6379`；Redis Stack 容器同时暴露 `8001` 端口供 RedisInsight 管理界面访问。
- **兼容说明**：课程环境允许的纯 Redis 2.8.9 不支持向量能力，向量相关功能必须使用 Redis Stack。

### 6.3 Elasticsearch

- **职责**：`search` 模块提供职位/简历全文检索、条件筛选、高亮、索引重建。
- **数据模型**：`JobDocument`、`ResumeDocument`（对应 `search/document` 包），通过 Spring Data Elasticsearch Repository 操作。
- **配置**：`spring.data.elasticsearch.uris=http://localhost:9200`（联调模式）。
- **版本**：容器使用 `elasticsearch:7.17.21`（`discovery.type=single-node`、`xpack.security.enabled=false`），兼容课程要求的 Windows 版 ES 7.13+。
- **数据来源**：简历的 `parsed_content` 字段与职位的 `description`/`requirements` 字段同步至 ES 索引。

### 6.4 Spring AI（大模型能力）

- **配置**：`recruitment-ai-service/application.yml` 中 `spring.ai.openai.api-key`/`base-url` 通过环境变量注入（`OPENAI_API_KEY`、`OPENAI_BASE_URL`），默认模型 `gpt-4o-mini`。
- **用途**：简历结构化解析、简历评估、人岗匹配解释生成、正式/模拟面试题生成、模拟面试评估。
- **提示词管理**：各业务模块下设 `prompt` 子包集中管理提示词模板，避免硬编码散落。

### 6.5 MySQL

- 版本 `mysql:8.0`，字符集 `utf8mb4`；通过 `docker-compose.yml` 挂载 `sql/init.sql` 作为容器首次启动初始化脚本。
- ORM 框架：MyBatis-Plus，`map-underscore-to-camel-case: true`，逻辑删除全局配置（`logic-delete-field: deleted`）。

## 7. 项目整体搭建思路

### 7.1 搭建顺序（自底向上）

```text
1. 基础 CRUD          用户、角色、职位、简历、投递、面试等表与实体
2. 登录鉴权与角色权限    Spring Security + JWT，四类角色权限隔离
3. 投递状态流转与面试分配 job_application 状态机、HR 分配面试官
4. ES 同步与搜索        简历/职位写入后同步 ES，实现全文检索
5. Redis Stack 向量与匹配 Embedding 生成、写入向量库、人岗匹配计算
6. Spring AI 深度集成    简历解析、正式出题、模拟面试、匹配解释
```

### 7.2 最小可运行闭环

```text
前端：Vue 3 + Element Plus + Pinia + Vue Router
核心服务：Spring Boot + Security + MyBatis-Plus + Redis + ES
AI 服务：Spring Boot + Spring AI + Redis Stack
服务治理：Nacos + OpenFeign
数据库：MySQL
```

### 7.3 环境搭建路径

1. **环境准备**：JDK 21+、Maven 3.9+、Node.js 18+、MySQL 8（必需）；Nacos/Redis Stack/ES（完整联调时需要）。
2. **数据库初始化**：执行 `sql/init.sql`，默认库 `smart_recruitment`。
3. **中间件启动**：`docker compose up -d`（`docker/docker-compose.yml`）一次性拉起 MySQL/Redis Stack/ES/Nacos，或手动安装本机中间件。
4. **后端启动**：
   - 本地轻量模式（`-Dspring-boot.run.profiles=local`，默认）：验证前后端联通，不依赖中间件。
   - 完整模式（`-Dspring-boot.run.profiles=dev`）：依赖已启动中间件，配置从 Nacos 动态拉取。
5. **前端启动**：`npm install && npm run dev`，通过 Vite 代理将 `/api` 转发至 `http://localhost:8080`。
6. **一键脚本**：`start-dev.ps1 -WithDocker -WithAiService` 自动完成中间件、核心服务、AI 服务、前端的启动编排。

### 7.4 关键设计原则

- **服务解耦**：AI 能力独立成服务，核心业务服务通过 Feign 调用，便于独立扩展与替换模型。
- **可渐进式联调**：默认 `local` Profile 保证开发不被中间件阻塞，`dev` Profile 承接完整联调，两套配置互不影响。
- **数据一致性**：简历/职位变更需同步 MySQL、ES、Redis Stack 三端，通过统一的 Service 层编排保证顺序与补偿。
- **权限强隔离**：AI 相关接口（正式出题 vs 模拟面试）在 Controller 与权限注解层面物理隔离，避免误调用。
