Redis：

| 应用场景            | 对应接口/模块                         | 具体用法                                                     |
| ------------------- | ------------------------------------- | ------------------------------------------------------------ |
| 高频数据缓存        | /api/job/list, /api/job/category/list | 职位与分类缓存：职位列表和分类数据是“读多写少”的。将热门职位列表或分类树缓存到 Redis 中，避免每次查询都击穿到 MySQL 数据库。 |
| AI 结果缓存（可选） | /api/ai/resume/parse                  | 接口幂等性与缓存：文档提到“相同简历内容的完整解析结果会在 AI 服务进程内缓存 15 分钟”。你可以用 Redis 替代进程内缓存，Key 为 AI:RESUME:MD5(文件内容)，Value 为解析结果。这样即使服务重启，缓存依然有效，且多实例部署时共享缓存。 |
| 数据统计            | /api/dashboard/stats                  | 实时计数：HR 工作台需要统计“今日投递数”、“今日面试数”。使用 Redis 的 INCR 命令进行实时计数，比直接查数据库 COUNT(*) 效率高得多。 |

ElasticSearch:

| 应用场景       | 对应接口/模块             | 具体用法                                                     |
| -------------- | ------------------------- | ------------------------------------------------------------ |
| 职位全文检索   | /api/search/job           | 多字段匹配：用户搜索“Java”时，不仅要匹配职位名称，还要匹配职位描述、技能要求、公司名称等。ES  支持分词和高亮显示，能实现“搜得到、搜得准”。 |
| 简历人才库搜索 | /api/search/resume        | HR 找人：HR 需要搜索“3年经验、本科、精通SpringBoot”的候选人。ES 支持复杂的组合查询（Bool  Query），能快速从海量简历中筛选出匹配的人才。 |
| 索引同步       | /api/search/rebuild-index | 数据一致性：当 MySQL 中的职位或简历数据发生变化（增删改）时，通过监听数据库变更或在业务代码中，同步更新 ES  中的索引数据。 |

nacos:

| 应用场景       | 对应模块               | 具体作用                                                     |
| -------------- | ---------------------- | ------------------------------------------------------------ |
| 统一配置中心   | 所有模块               | 管理配置文件：将 application.yml 中的数据库连接、Redis 地址、ES 地址、文件上传路径等配置移到 Nacos 控制台。修改配置后，服务无需重启即可生效（热更新）。 |
| AI 密钥管理    | recruitment-ai-service | 安全管理 API Key：文档提到 application-secrets.properties 包含 API Key。千万不要把 Key 硬编码在代码里！ 应该将其配置在 Nacos 中，并设置权限，确保只有 AI 服务能读取，保障密钥安全。 |
| 服务注册与发现 | 微服务调用             | 服务间通信：你的系统分为 recruitment-service (8080) 和 recruitment-ai-service (8081)。主服务需要调用 AI 服务时，不需要写死 http://localhost:8081，而是通过 Nacos 发现 AI 服务的地址，实现负载均衡和故障转移。 |
